# score-app Skill 重设计方案

**日期**: 2026-04-14  
**背景**: 根据马拉松组委会 Q&A（Q7/Q8/Q9）重新设计评分维度与报告格式

---

## 背景与目标

score-app skill 的用途是：**候选人在考试过程中对自己的项目进行自主评分**，逆向推导组委会评审规则，找到失分点并优先修复。

关键设计原则：
- 权重完全公开（帮助候选人计算 ROI）
- 报告采用"教练视角"：不只是评价，而是告诉候选人"先做什么"
- 评分必须基于代码证据，不能主观臆断

---

## 评分维度（5 维度，替代原 7 维度）

### 变更说明

| 原维度 | 新维度 | 变更原因 |
|--------|--------|---------|
| 功能完整性（30%） | 功能完整性与需求理解（30%） | Q9：需求理解本身是考察维度，合并进来 |
| 技术实现质量（20%）+ 代码可维护性（15%） | 代码质量与工程规范（25%） | 两者概念重叠，合并；权重下调5%让给可测试性 |
| 可测试性（15%） | 可测试性（20%） | Q8 明确列为第3优先级，权重上调 |
| 设计文档与Spec（10%）+ 开发与使用文档（5%） | 文档质量（15%） | 合并为一个维度，总权重不变 |
| AI 工具链使用（5%） | AI 工具合理使用（10%） | Q7 明确列出，"合理"是关键词；权重上调 |

### 权重汇总

| 维度 | 权重 | 对应 Q8 优先级 |
|------|------|--------------|
| 功能完整性与需求理解 | 30% | 第1位（基础分关键） |
| 代码质量与工程规范 | 25% | 第2位（可读性/规范/错误处理） |
| 可测试性 | 20% | 第3位（关键逻辑有测试覆盖） |
| 文档质量 | 15% | 第4位（README + 设计方案） |
| AI 工具合理使用 | 10% | Q7 并列维度 |

## 各维度评分标准

### 1. 功能完整性与需求理解（30%）

评估内容：功能实现数量 + 需求理解深度

检查点：
- 所有核心功能是否有对应实现（对照需求文档逐条核验）
- 边界情况和异常路径是否处理（而非只有 happy path）
- 对模糊需求是否做了合理假设并在设计文档中说明（Q9 考察点）
- 交互设计是否合理（页面流程符合用户预期）

| 分段 | 标准 |
|------|------|
| 90–100 | 核心+扩展功能均完成，边界情况处理完善，模糊需求有记录假设 |
| 70–89 | 主要功能完成，少量边界场景或次要功能缺失 |
| 50–69 | 超过一半功能实现，有明显缺失模块 |
| 0–49 | 核心功能缺失，无法满足基本需求 |

### 2. 代码质量与工程规范（25%）

评估内容：技术决策合理性 + 代码可读性 + 安全性

检查点：
- RESTful API 设计规范（HTTP 方法语义、状态码、URL 命名）
- 错误处理完整性（后端统一异常 / 前端 loading/error/empty state）
- 命名规范和代码可读性（无魔法数字、语义明确）
- 架构分层清晰（controller/service/repository 职责不越界）
- 无安全隐患（无硬编码密码/密钥、SQL 注入防护、鉴权逻辑正确）
- 无明显重复代码
- 架构优化为加分项，不是基础要求（Q8 最低优先级）

| 分段 | 标准 |
|------|------|
| 90–100 | 设计严谨，实现规范，无明显安全问题 |
| 70–89 | 整体质量较高，有少量可优化点 |
| 50–69 | 基本可用，存在明显设计问题或安全隐患 |
| 0–49 | 严重技术缺陷或安全漏洞 |

### 3. 可测试性（20%）

评估内容：测试数量 × 测试质量

检查点：
- 后端：单元测试/集成测试覆盖的业务场景数量
- 前端：组件测试或 E2E 测试
- 测试用例质量：包含边界场景和异常路径，不只是 happy path
- 核心业务逻辑（非 getter/setter）是否有覆盖

| 分段 | 标准 |
|------|------|
| 90–100 | 覆盖全面，含边界+异常场景 |
| 70–89 | 覆盖主要功能，测试用例质量较好 |
| 50–69 | 仅少量测试，覆盖面有限 |
| 0–49 | 无测试或测试形同虚设 |

### 4. 文档质量（15%）

评估内容：README 可用性 + 设计文档完整性

检查点：
- README：项目简介、环境依赖、快速启动步骤（新人能独立跑起来）
- 设计文档：架构设计、数据库设计、API 设计（doc/ 目录）
- 对模糊需求的假设是否有文字记录（与第1维度联动）
- 文档与实际实现是否一致（有文档但描述错误不得分）

| 分段 | 标准 |
|------|------|
| 90–100 | 文档完善，新人可按文档独立运行，设计文档与实现高度一致 |
| 70–89 | 基本文档，覆盖主要设计决策 |
| 50–69 | 文档存在但简略或与实现有出入 |
| 0–49 | 无 README 或完全无法依赖文档运行 |

### 5. AI 工具合理使用（10%）

评估内容：AI 工具的使用深度与合理性（"合理"是关键词，Q7 原文）

检查点：
- CLAUDE.md / AGENTS.md 内容是否有实质内容（而非复制模板）
- `.agents/` 目录是否有项目专属自定义 Skill
- 提交记录是否有 AI 辅助开发的迹象
- 合理性：AI 用于实质性任务（而非只生成样板代码）

| 分段 | 标准 |
|------|------|
| 90–100 | AI 工具链配置完善，有明显提效的使用证据 |
| 70–89 | 有基本 AI 配置，内容有实质内容 |
| 50–69 | 有 AI 配置文件但内容简略或为模板 |
| 0–49 | 无 AI 工具链使用证据 |

## 报告格式

报告风格：教练视角，三层结构：
1. **总览**：ASCII 进度条 + 诊断句（先做什么可以到多少分）
2. **维度明细**：每个维度的亮点/失分项，引用文件路径和行号作为证据
3. **行动清单**：按涨分 ROI 排序（涨分幅度 / 工作量）

### 总览格式

```
┌─────────────────────────────────────────┐
│  综合得分：  78.5 / 100                  │
│                                         │
│  功能完整性  ████████░░  82  × 30% = 24.6│
│  代码质量    ███████░░░  74  × 25% = 18.5│
│  可测试性    █████░░░░░  55  × 20% = 11.0│
│  文档质量    ████████░░  80  × 15% = 12.0│
│  AI 工具     ████████░░  84  × 10% = 8.4 │
└─────────────────────────────────────────┘
```

诊断句示例：
> 主要失分在可测试性（-9分）和代码质量（-6.25分）。优先补测试，其次修代码质量问题，预计可提升至 87+ 分。

### 行动清单格式

| 优先级 | 行动 | 预计涨分 | 工作量 |
|--------|------|---------|--------|
| 🔴 高  | 补后端集成测试（覆盖主要 API + 边界场景） | +8~10 分 | 中 |
| 🔴 高  | 修前端 loading/error 状态 | +3~4 分 | 低 |
| 🟡 中  | 记录需求模糊点的设计假设到 doc/ | +2~3 分 | 低 |
| 🟡 中  | 修复 HTTP 状态码规范问题 | +2 分 | 低 |
| 🟢 低  | 补充前端组件测试 | +2~3 分 | 高 |

---

## 评分流程

1. **读取需求文档**：提取所有功能需求点，建立检查清单
2. **并行扫描项目**：同时扫描 backend/、frontend/、doc/ 和根目录
3. **逐维度评分**：0–100 整数分，每条结论引用文件路径+行号
4. **计算总分**：加权总分 = Σ（各维度原始分 × 权重）
5. **生成报告**：保存到 `doc/evaluation-report.md`
