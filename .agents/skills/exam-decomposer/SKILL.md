# 马拉松考试题目拆解

本 skill 帮助在竞赛/考试类项目中快速将需求文档转化为结构化的开发任务清单，让你少走弯路、快速动手。

## 工作流程

### Step 1：定位需求文档

优先读取以下文件（按优先级）：
1. `题目描述.md`
2. 用户明确指出的文件
3. `README.md` 中包含需求描述的部分

同时读取 `认证指引.md`（若存在），了解评分维度和工程结构要求——这对后续任务优先级的判断很有价值。

### Step 2：需求分析与拆解

通读需求文档，结构化提取：

- **背景与痛点**：当前系统/流程有什么问题，为什么要做这个系统
- **核心业务流程**：主流程是什么（通常以流程图或步骤描述呈现）
- **功能清单**：按原文列出所有功能点，不遗漏、不臆测

### Step 3：提炼核心业务目标

用 1-2 句话回答：这个系统要解决什么问题，给谁带来什么价值。

这句话是整个 TODO.md 的"北极星"，帮助在开发时判断什么是主干、什么是加分项。

### Step 4：拆分功能模块

将功能点按领域聚合为独立模块，每个模块描述：
- 模块名称
- 包含的功能列表
- 预估后端 API 数量
- 对应的前端页面/组件

常见模块示例：用户认证、资源管理、核心业务（如预约/订单）、查询与筛选。

### Step 5：梳理关键业务规则

重点关注这些容易被遗漏、但评分时会被检查的规则：

- **冲突检测**：哪些操作必须检查冲突（时间冲突、资源占用冲突等）
- **数据约束**：字段范围、时间限制、数量上限
- **权限规则**：谁能操作什么
- **边界情况**：历史数据、空集、极限值等

每条规则需明确对应哪个模块和 API，避免实现时遗漏。

### Step 6：识别不明确点，形成待确认问题列表

审视需求中模糊、缺失或有歧义的地方，分两类记录：

**阻塞性问题**：不确认就无法设计数据模型或 API 的决策
**可假设问题**：可先做合理假设推进开发，后续补充说明

格式：
```
? [模块] 具体问题
  → 假设：如有合理假设，写出并说明理由
```

对于"可假设"问题，主动给出假设方案，避免等待阻塞开发进度。

### Step 7：创建 TODO.md

在项目根目录创建 `TODO.md`，使用以下结构：

```markdown
# 项目开发 TODO

> 核心目标：[一句话描述]

## 模块总览

| 模块 | 功能数 | 预估 API 数 | 优先级 |
|------|--------|-------------|--------|
| ...  | ...    | ...         | ...    |

## 开发任务

### 基础设施
- [ ] 数据库初始化 & 实体设计（P0）
- [ ] 项目脚手架搭建（P0）

### [模块名]
- [ ] 任务描述 [后端/前端]（P0/P1/P2）

## 待确认问题

- [ ] [阻塞] 问题描述
- [ ] [可假设] 问题描述 → 假设：...

## 关键业务规则备忘

- 规则 1（对应模块/API）
- 规则 2（对应模块/API）
```

## 优先级定义

| 级别 | 含义 |
|------|------|
| P0 | 核心流程，必须完成，否则无法演示 |
| P1 | 重要功能，考核重点，尽力完成 |
| P2 | 加分项，时间允许再做 |

## 输出规范

分析完成后，在对话中依次展示：
1. **核心目标**（1-2 句）
2. **功能模块总览**（表格）
3. **关键业务规则**（重点列出容易遗漏的）
4. **待确认问题列表**（区分阻塞/可假设）
5. 告知 TODO.md 已创建，路径为项目根目录

不要逐字复述需求文档，聚焦在提炼、归纳、补全和风险识别上。
